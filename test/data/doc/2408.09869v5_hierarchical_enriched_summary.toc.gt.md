## Docling Technical Report [reference=#/texts/1] (summary=Docling is a lightweight, MIT‑licensed open‑source package that converts PDF documents into structured formats. It leverages state‑of‑the‑art AI models—DocLayNet for layout analysis and TableFormer for table structure recognition—to achieve high accuracy while running efficiently on commodity hardware with minimal resources. The modular code interface makes it straightforward to extend the tool or integrate new models and features.)

### Version 1.0 [reference=#/texts/2] (summary=Version 1.0 lists the contributors—Christoph Auer, Maksym Lysak, Ahmed Nassar, Michele Dolfi, Nikolaos Livathinos, Panos Vagenas, Cesar Berrospi, Ramis Matteo Omenetti, Fabian Lindlbauer, Kasper Dinkla, Lokesh Mishra, Yusik Kim, Shubham Gupta, Rafael Teixeira de Lima, Valery Weber, Lucas Morin, Ingmar Meijer, Viktor Kuropiatnyk and Peter W. J. Staar.  
They are affiliated with the AI4K Group at IBM Research Rüschlikon in Switzerland.)

### Abstract [reference=#/texts/5] (summary=Docling is a lightweight, MIT‑licensed open‑source tool that converts PDF documents into structured formats using advanced AI models. It leverages DocLayNet for layout analysis and TableFormer for table structure recognition, achieving efficient performance on commodity hardware with a modest resource footprint. The package’s code interface is designed for easy extensibility, allowing users to add new features and models seamlessly.)

### 1 Introduction [reference=#/texts/7] (summary=PDFs are hard to parse because they’re optimized for printing rather than structure, making open‑source extraction tools lag behind commercial solutions even as LLMs make content extraction more valuable. Docling fills this gap with a lightweight Python library that uses advanced layout and table recognition models to convert PDFs into fast JSON or Markdown, recover metadata, figures, tables, and optionally apply OCR. It supports both high‑throughput batch processing and interactive use, and can leverage GPU/MPS accelerators for efficiency.)

### 2 Getting Started [reference=#/texts/19] (summary=Docling can be installed from PyPI and includes all model assets that are automatically downloaded to a Hugging Face cache on first use (or pre‑installed if desired). It offers simple APIs for converting PDFs from files, URLs or binary streams into JSON or Markdown—both single documents and batches—while allowing optional configuration of pipeline features such as OCR or table recognition, input size limits, and CPU thread budgets. A Dockerfile is also provided for containerized deployment, and further examples are available in the GitHub repository.)

### 3 Processing pipeline [reference=#/texts/25] (summary=Docling parses PDFs to obtain text tokens and page images, then runs a customizable sequence of AI models—currently an RT‑DETR‑derived layout detector and the TableFormer table recognizer—to annotate each page; the outputs are aggregated and post‑processed to infer metadata, language, reading order, and assemble a structured document object that can be exported as JSON or Markdown. The PDF backend is open‑source (docling‑parse using qpdf) with an optional pypdfium fallback, supports optional OCR via EasyOCR, and allows community extensions through a BaseModelPipeline abstraction.)

#### 3.1 PDF backends [reference=#/texts/27] (summary=Docling’s PDF backend interface is designed to (a) extract all text along with its geometric coordinates and (b) render each page exactly as it would appear in a viewer. Existing Python libraries failed to meet these needs—pymupdf was limited by restrictive licensing, while pypdfium and PyPDF suffered from slow performance or merged‑cell quality issues. To address this, Docling offers multiple backends: a custom parser built on the low‑level qpdf library (in the separate docling-parse package) as the default, with an optional pypdfium backend for cases where font‑encoding problems arise.)

#### 3.2 AI models [reference=#/texts/49] (summary=Docling opens two powerful AI models—an RT‑DETR‑derived layout analyzer that predicts page element boxes and a TableFormer vision‑transformer that reconstructs complex table structures—both pretrained on HuggingFace and run via ONNX runtime or PyTorch. The pipeline processes 72 dpi page images in sub‑second CPU time, post‑processes overlapping detections into coherent units, feeds detected tables to TableFormer for row/column structure recovery (2–6 s per table), and optionally runs EasyOCR on high‑resolution scans (≈30 s/page). The project invites community contributions to add faster OCR backends and further extend the open‑source ecosystem.)

#### Layout Analysis Model [reference=#/texts/51] (summary=The layout analysis model is an RT‑DETR–based object detector trained on DocLayNet and proprietary data, producing bounding boxes and classes for page elements. It runs with ONNX Runtime on a single CPU at 72 dpi, achieving sub‑second latency; the predictions are filtered to remove overlaps and then aligned with PDF text tokens to assemble coherent units such as paragraphs, titles, lists, captions, figures, or tables.)

#### Table Structure Recognition [reference=#/texts/54] (summary=TableFormer is a vision‑transformer model that predicts the logical row and column structure of tables, classifying cells as column headers, row headers or body while handling missing borders, empty cells, spans, hierarchy, and alignment issues. In Docling it receives cropped table images and their extracted text, then maps its predictions back to the original PDF cells to avoid costly re‑transcription; processing typically takes 2–6 seconds per table on a standard CPU.)

#### OCR [reference=#/texts/58] (summary=Docling optionally supports OCR, currently using the EasyOCR library on 216‑dpi images to capture fine print in scanned PDFs or bitmap pages. While EasyOCR provides reasonable transcription quality, it can be slow—upwards of 30 seconds per page on CPU. The project invites open‑source contributors to add additional OCR backends and improve performance.)

#### 3.3 Assembly [reference=#/texts/61] (summary=Docling aggregates page‑level predictions into a structured document object defined by the docling-core package. A post‑processing model enriches this object—detecting language, correcting reading order, linking figures to captions, and labeling metadata such as title, authors, and references. The enriched document can then be exported as JSON or converted to Markdown.)

#### 3.4 Extensibility [reference=#/texts/63] (summary=Docling lets users extend its central model pipeline—between parsing and output assembly—by subclassing `BaseModelPipeline` or cloning the default, enabling full customization of model chains, replacement/addition of models, and new configuration parameters; a custom pipeline class can be passed to the main conversion functions.  
Model implementations must implement `__call__`, accepting an iterator over page objects and returning an augmented iterator with added predictions.  
We invite community contributions of additional or alternative models and improvements.)

### 4 Performance [reference=#/texts/66] (summary=Docling was benchmarked on a 225‑page mix of arXiv papers and IBM Redbooks using default settings, two PDF backends (native and pypdfium), and thread budgets of 4 and 16 on an Apple M3 Max and an Intel Xeon E5‑2690. The native backend takes 177–375 s (1.27–0.60 pages/s) while consuming ~6 GB, whereas the pypdfium backend cuts runtime to 92–239 s (2.18–1.57 pages/s) and memory to 2.5–2.4 GB, but may sacrifice table‑structure quality; performance improves with more threads on both systems. GPU acceleration for the AI models is under development and may become available when CUDA is detected by ONNXRuntime/torch.)

### 5 Applications [reference=#/texts/73] (summary=Docling’s high‑quality, richly structured conversion supports a wide range of downstream tasks such as enterprise search, passage retrieval, classification and knowledge extraction that can treat tables, figures, sections, and references separately. Its open‑source “quackling” package enables document‑native vector embedding and chunking for retrieval‑augmented generation (RAG), integrating smoothly with LLM frameworks like LlamaIndex. Because it’s fast, stable, and inexpensive, Docling is ideal for building document‑derived datasets—especially with its powerful table recognition—for automated knowledge‑base construction and large‑scale multimodal training via IBM’s data prep kit.)

### 6 Future work and contributions [reference=#/texts/75] (summary=Docling is designed for easy extension of its model library and pipelines, with plans to add new models such as a figure‑classifier, equation‑recognition, code‑recognition, and others to improve conversion quality and enrich metadata extraction. Future work also targets better GPU acceleration, testing, and an improved Docling‑native PDF backend. The project welcomes community contributions under the MIT license and encourages citing this technical report when using Docling.)

### References [reference=#/texts/78] (summary=The references compile a set of open‑source tools—EasyOCR for multilingual OCR, PyTorch 2 and ONNX Runtime for accelerated inference, and PDF libraries such as qpdf, PyMuPDF, pypdf, and PyPDFium2—that underpin document processing pipelines. They also point to large annotated datasets (Doclaynet, Patcid) and transformer‑based models (Tableformer, LlamaIndex) tailored to layout analysis and table structure recognition. Recent research highlighted includes optimized table tokenization, ESG KPI extraction from tables via large language models, and a performance comparison of DETR versus YOLO for real‑time object detection.)

### Appendix [reference=#/texts/98] (summary=The appendix presents Docling’s Markdown and JSON outputs while highlighting that PDF document conversion remains hard because layouts, languages, and formats vary so widely that a single ML model cannot easily generalise. To build a reliable benchmark, the authors introduced pixel‑accurate “snapping” boxes for text segments, rejected un‑annotatable pages, and achieved human mAP scores of 84–94 % across classes, which serve as an unbiased baseline. Baseline object‑detection models (Mask R‑CNN, Faster R‑CNN, YOLOv5x6) trained on DocLayNet reach mAPs of 68–88 %, with performance plateauing once ~80 % of the dataset is used, indicating that further data alone will not markedly improve results.)

### 5 EXPERIMENTS [reference=#/texts/699] (summary=DocLayNet evaluates object‑detection baselines (Mask R‑CNN, Faster R‑CNN, YOLOv5) using COCO mAP@0.5–0.95 on 1025×1025 RGB images, showing that the learning curve flattens once ~80 % of the data is used.  
The models achieve mAP scores only 6–10 % below the inter‑annotator agreement, indicating a meaningful challenge, while YOLOv5x even surpasses human performance on abundant classes such as Text, Table and Picture; Mask R‑CNN and Faster R‑CNN perform comparably, suggesting pixel‑level segmentation offers limited benefit over bounding boxes.)

#### Baselines for Object Detection [reference=#/texts/704] (summary=Baseline experiments on Mask R‑CNN, Faster R‑CNN and YOLOv5 (all trained/evaluated on 1025×1025 RGB images) show only a small mAP gap between the models, yet all are roughly 6–10 % below the human triple‑annotator mAP. The similarity of Mask R‑CNN and Faster R‑CNN scores indicates that pixel‑level segmentation derived from bounding boxes does not add benefit, while YOLOv5x excels—surpassing humans on frequent, visually distinctive classes such as Text, Table, and Picture. These results highlight DocLayNet’s difficulty in bridging the gap between human and machine performance.)
